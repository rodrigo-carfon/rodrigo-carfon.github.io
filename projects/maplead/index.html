Turning an unstructured
public source into a
scored, schema'd dataset.

MapLead takes a plain search — a business category and a city — and returns a structured dataset: 14 fields per business, a sample of reviews, the marketing pixels each one runs, and a transparent 0–100 score. Playwright does the reading, a heuristic does the ranking, and the whole thing runs async on a free-tier box that gives it 512 MB to work with.

architecture

Five stages, end to end.

Every run is the same pipeline. A FastAPI endpoint validates the request, hands the job to a background worker and returns immediately; the worker walks the five stages and mails the result out when it is done.

The contract

One endpoint. The 202 is the interesting part: the response is an acknowledgement, not a result.

POST /scrape 
# request
{
  "query":          "dental clinics miami",   # 2–200 chars
  "max_businesses": 5,                        # 1–100
  "max_reviews":    5,                        # 0–100
  "email":          "you@example.com"
}

# response — immediately, before any scraping happens
202 Accepted
{ "status": "processing", "query": "dental clinics miami", … }

Decisions & constraints

Most of the design here was forced by the box it runs on. That turned out to be the useful part.

async, not
request/response

A run takes minutes, not milliseconds. The endpoint validates, queues the job on FastAPI's BackgroundTasks and answers 202 straight away — holding an HTTP request open for five minutes would be timed out by every proxy between here and the client. The result arrives by email because email is a queue someone already checks.

email over HTTP,
not SMTP

The free tier blocks outbound SMTP connections, so delivery goes through Brevo's HTTP API instead. A hosting constraint quietly picked the transport.

recycle the browser
every 3 businesses

Chromium's memory footprint grows across a run. On a 512 MB box that is the difference between finishing and being OOM-killed halfway, so the browser is torn down and rebuilt every third business. Low-resource Chromium flags are on in Docker and off on a desktop, where --single-process crashes it.

skip, don't hang

A 90-second ceiling per business, set from a benchmark where the typical page took ~12 s and the worst case landed near 15 s. Past the ceiling the business is skipped rather than allowed to block the job — one unresponsive profile should cost you one row, not the run.

demo capped
at 5 × 5

The API's hard limits are 100 businesses and 100 reviews. The public demo below pins both to five, so a stranger's run stays inside what the free tier can serve without pushing everyone else into a cold start.

Pixel detection

Each business site is fetched and its HTML matched against nine detectors, each keyed on specific inline-install signatures rather than a loose brand string — the point is to avoid false positives, not to catch every exotic install.

The 0–100 heuristic

Four additive rules, evaluated as each business is inserted. Deliberately transparent — every score can be read straight back off the row that produced it.

has website+30
has phone+25
rating ≥ 4.0+30
reviews ≥ 50+15
0 — 40 · hot lead
Little to no online presence. The gap is the opportunity.
41 — 70 · established
Basic presence, with obvious room to improve.
71 — 100 · benchmark
Digitally strong — a reference point, or a competitor to watch.

The score is what turns a raw scrape into something ordered. Sort ascending and you get a worklist; sort descending and you get the segment's benchmarks. Same data, one column.
