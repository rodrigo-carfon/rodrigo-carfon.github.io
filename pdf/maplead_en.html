05 · architecture

How it works

A single FastAPI service handles everything. A POST request triggers a background task — the API responds immediately with 202 Accepted while Playwright drives a stealth Chromium through Google Maps. Results are shaped with pandas into two CSVs and delivered via Gmail SMTP.

Memory constraints on the free tier

The biggest engineering challenge was fitting a headless Chromium into Render's 512MB free-tier container. A few deliberate choices made it work:

  • Images, fonts and media aborted at the network layer via Playwright route() — not the engine flag, which breaks the Maps feed
  • Browser recycled every 3 businesses to release Chromium's accumulated memory
  • --single-process flag on Linux only (crashes Chromium on Windows)
  • Pixel detection runs as concurrent HTTP GETs via APIRequestContext — no extra browser tab
06 · engineering challenges

Problems solved along the way

The interesting part wasn't the happy path. Each item below was a real constraint or bug hit during development.

problem

Duplicate reviews on every run

Google Maps reuses the same data-review-id across nested DOM elements, and the overview tab only previews ~3 reviews — naive scraping capped every business at three duplicated rows.

solution

Dedup by stable ID + full reviews tab

Always navigate to the full Reviews tab, then deduplicate on the stable review ID while scrolling until max_reviews unique reviews load.

problem

Crashes only on Windows

Under uvicorn on Windows, Playwright threw NotImplementedError — the SelectorEventLoop can't spawn the Node driver subprocess.

solution

Dedicated ProactorEventLoop per thread

The scraper runs in its own thread with a ProactorEventLoop (supports subprocesses), isolated from uvicorn's loop — Windows-only, transparent on Linux.

problem

One slow listing hangs the whole job

A single broken Maps listing would stall the entire background task — and a hung background task means no email ever arrives.

solution

Statistical per-business timeout

Benchmarked per-business time (mean + 2σ) and wrapped each listing in asyncio.wait_for(). Slow listings are skipped, not fatal — the run always finishes.

problem

Business name came back blank intermittently

Google Maps renders the <h1> element immediately but fills its text via JS shortly after — on the slow free-tier CPU, reading the element too early returned an empty string.

solution

Wait for non-empty text content

Added a wait_for_function that polls until h1.textContent.trim().length > 0 before reading — tolerant of slow CPU without a fixed sleep.
