Portfolio Project · Web Scraping & Automation

MapLead
Intelligence

A web-scraping pipeline that searches Google Maps for local businesses, collects their listings and individual reviews, scores their digital presence, and emails the structured results back as ready-to-use CSVs. One HTTP request triggers it all — packaged as a single Docker container on a free-tier host.

01 · context

The problem this solves

Marketing agencies, consultants, and growth teams constantly need structured data on local businesses — to prospect clients, map competitors, or audit digital presence. Doing it by hand means opening each Google Maps listing one by one, copy-pasting names, ratings, phones, and websites into a spreadsheet, then repeating it for dozens of businesses. Slow, inconsistent, impossible to scale.

✗

The manual way

Open each listing, copy data one by one, fragmented spreadsheets. Hours to map 20 businesses — no standardization, no reviews at scale, no signal on who actually needs help.

✓

With MapLead

Fill one form. Minutes later, two clean CSVs land in your inbox: every business with a computed Digital Presence Score and tracking-pixel detection, plus every individual review with its own rating — ready for analysis.

03 · architecture

How it works

The whole system is a single self-contained service. A form (or any HTTP client) posts a search request to a FastAPI endpoint, which answers 202 Accepted immediately and runs the scrape in the background. Playwright drives a stealth Chromium through Google Maps; the results are shaped with pandas into two CSVs and emailed via SendGrid.

Deployment

One container, no external services. It ships as a single Docker image and deploys to Render's free tier — HTTPS out of the box, automatic builds on every push, and deliberately kept lean enough to fit the smallest free container that exists.

maplead scraper · API · mailer
python:3.11-slimplaywright chromiumfastapi + uvicorn binds $PORT--single-process

Exposes POST /scrape and GET /health. Rebuilt automatically on every push, and tuned to run entirely within a free tier's memory budget.

Triggering a collection

A single request kicks off everything. The API validates the input, returns instantly, and does the work in the background.

example request · POST /scrape 
# request body
{
  "query":          "barbershops campinas",
  "max_businesses": 15,   # 1–15
  "max_reviews":    5,    # 0–5 per business
  "email":          "you@example.com"
}

# immediate response — work continues in the background
202 Accepted
{
  "status":  "processing",
  "message": "Search started! Results will be emailed in a few minutes."
}

# ~minutes later → businesses.csv + reviews.csv arrive by email

The Digital Presence Score

The score is the project's core idea: a simple, transparent rule that turns four raw fields into a single number, so a marketing team can instantly sort a list by who is digitally behind — and therefore who is worth a call.

has website+30 pts
has phone+25 pts
rating ≥ 4.0+30 pts
50+ reviews+15 pts
0 — 40 · hot lead
No website or phone. Prime opportunity for digital marketing services.
41 — 70 · established
Basic presence in place. Clear room for improvement.
71 — 100 · benchmark
Digitally dominant in the niche. A reference point for the segment.
05 · engineering journey

Discoveries & problems solved

The interesting part of this project wasn't the happy path — it was everything that fought back. Each item below was a real bug or constraint hit during development, and the fix is what lets the project actually run in production on the smallest free tier available.

problem

Reviews came back duplicated

Google Maps reuses the same data-review-id across several DOM nodes, and the listing's overview tab only previews ~3 reviews — so naive scraping capped every business at three duplicated rows.

solution

Dedup by stable id + open the real tab

Always click into the full Reviews tab, then deduplicate on the stable review id while scrolling until max_reviews unique reviews are loaded.

problem

The server crashed only on Windows

Under uvicorn on Windows, Playwright threw NotImplementedError: the SelectorEventLoop can't spawn the Node driver subprocess it needs.

solution

A dedicated event loop in a thread

Run the scraper in its own thread on a ProactorEventLoop (which supports subprocesses), isolated from uvicorn's loop — Windows-only, transparent on Linux.

problem

Fitting Chromium into a tiny free container

A headless Chromium plus the app overran the free tier's memory budget. Disabling images via the engine flag was the obvious fix — but it silently broke the Maps results feed.

solution

Block images at the network layer

Abort image/font/media requests with a route handler (the engine still believes images are on) plus --single-process. Measured real usage by USS, not summed RSS.

problem

One stuck listing could hang the whole job

A single slow or broken Maps listing would stall the entire run — and for a background email job, a hang means no email ever arrives.

solution

A statistical per-business timeout

Benchmarked per-business time and set a cap of max + 2σ. Anything slower is skipped, not fatal — so the run always finishes and the email always sends.

06 · live demo

Try it on a real search

Run the actual pipeline against a query of your choice. Your request hits the live API (hosted on a free tier, so the first call may take a few seconds to wake it up), which scrapes Google Maps in the background and emails you the two CSVs — the same files described above, generated on demand.

  • Pick any business type + city (e.g. barbershops campinas)
  • Each run collects 5 businesses with up to 5 reviews each
  • Be patient: it runs on a free-tier server, so the email can take ~5 minutes to arrive — that's normal, it is working
  • The email often lands in spam — check there too
⚙ Fixed scope: 5 businesses × 5 reviews per run — kept small so the free tier stays responsive.
